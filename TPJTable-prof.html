2.2. Analyser le code

Analyser le code fourni :

  1. Structure générale des classes

    1. Le code source des classes graphiques (initialisées avec W.B. puis modifiées par l’équipe projet) est structurée de la manière suivante :

      1. Les attributs « métiers de la classe »

      2. Les attributs graphiques (générés par W.B.)

      3. Les méthodes publiques (services offerts par la classe)

      4. Les méthodes privées gérées par W.B. (initialisation et accesseurs)

      5. Les méthodes privées liées aux fonctionnalités techniques du composant.

    2. Pour faciliter la compréhension de vos propres développements dans le cadre de ce projet, il est fortement conseillé de respecter cette classification pour les attributs et les méthodes que vous ajouterez aux composants existants.

  2. Main

    1. Contient le main() de lancement de l’appli.

    2. NB : Noter que vous NE DEVEZ PAS LA MODIFER dans le TP.

  3. Student

    1. Simple structure de données décrivant un étudiant (pourrait provenir d’un ex projet, par exemple pour la gestion des projets tutorés). Les données pourraient provenir d’une base de données

    2. Le Champ « id » serait l’identifiant unique de chaque student. Il sera géré par une base de données et nous ne l’utiliserons pour l’instant que par soucis de compatibilité future (sa valeur n’a pas d’importance).

    3. NB : Nous utilisons des attributs « public » pour des raisons de simplicité. Ce choix serait discutable dans le cadre d’un projet industrialisé.

    4. NB : Noter que vous NE DEVEZ PAS LA MODIFER dans le TP.

  4. DataLoader

    1. Classe en charge du chargement des données. En théorie depuis une base de données ou un fichier xml (ça vous rappelle des choses ?). Ici un simple bouchon de test.

    2. NB : Noter que vous NE DEVEZ PAS LA MODIFER dans le TP.

  5. StudentDialog

    1. Boite de dialogue utilisée pour créer ou modifier une donnée de type Student ainsi que pour confirmer la suppression d’une telle donnée.

    2. La classe dérive de JDialog et ne possède que 2 méthodes publiques :

      1. Le constructeur (en association avec une fenêtre appelante)

      2. La méthode ouvrirDialogue()

    3. L’appel de la méthode ouvrirDialogue() est bloquant pour le programme appelant (ouverture modale de boite de dialogue)

    4. Il existe 3 modes d’ouverture paramétrés par l’argument « mode » de la méthode ouvrirDialogue().

    5. Quel que soit le mode d’ouverture, le programme appelant récupèrera :

      1. Le code de retour de la méthode (true si l’action est confirmée)

      2. L’objet Student concerné (éventuellement modifié par l’utilisateur).

    6. NB : Cette boite de dialogue a donc pour rôle de manipuler une donnée de type Student. Elle ne doit réaliser aucune action directe sur l’interface graphique appelante ou sur la base de données.

    7. NB : Noter que vous NE DEVEZ PAS LA MODIFER dans le TP.

  2. On puisse modifier (entrer en modif) les colonnes nom, prénom, groupe directement sur chaque ligne du JTable. Mais pas la colonne Id. Modifier la méthode isCellEditable(). Dans quelle classe ?

  3. Modifier les valeurs des colonnes nom, prénom, groupe sur chaque ligne du JTable. Modifier la méthode setValueAt(). Dans quelle classe ? Avant modification, cette méthode vérifiera :

5. Pour aller plus loin

Vous pouvez essayer de programmer un renderer pour la colonne groupe pour formatter le texte ainsi : "1 - A". Chercher dans la documentation en ligne et regarder du côté de getCellRenderer() de la classe JTable ainsi que créer une sous-classe de DefaultTableCellRenderer.

Vous pouvez aussi programmer un editor pour la colonne groupe afin de mettre en place une liste déroulante de saisie avec les 8 valeurs possibles.
